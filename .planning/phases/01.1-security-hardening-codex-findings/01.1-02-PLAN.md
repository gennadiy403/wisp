<objective>
Fix HIGH SEC-03 race conditions в retry mechanism.

**Делегируется Codex** через `codex exec` — concurrency/lock логика, где Codex с xhigh reasoning ловит subtle bugs лучше.

**Two issues:**

1. **Single-flight retry (HIGH):** govori.py:1146-1157 `_hud_click_action` инкрементирует `_retry_count` без lock и не проверяет, не запущен ли уже worker. Повторные клики = parallel `_retry_transcription` threads на том же `_retry_buffer` → дубли paste, дубли API-запросов.

2. **Mode-aware retry (MEDIUM):** govori.py:1259 `_retry_transcription` всегда зовёт `paste_text(text + " ")`. Failed note-mode retry должен сохраняться через `save_or_merge_note()`; failed predict-mode retry должен переоткрыть predict UI; failed auto-send должен auto-send. Текущая реализация ломает UX этих режимов.

**Что делать (Codex):**
1. Добавить `_retry_in_progress` (bool) и `_retry_mode_snapshot` (dict с контекстом) в module-level state
2. В точке где буфер кладётся в retry (см. `stop_and_transcribe`, `_note_pipeline_background`) — снимок текущего mode (note_mode, predict_mode, auto_send, etc.)
3. `_hud_click_action`: с lock'ом проверить `not _retry_in_progress` И счётчик; set `_retry_in_progress=True`; стартовать thread
4. `_retry_transcription`: после success/fail в finally установить `_retry_in_progress=False` под lock; success диспатчить по сохранённому mode
5. Race-test mentally: сценарии "click during transcribing", "click while previous worker just finished but не успел очистить state", "fatal error mid-retry"
