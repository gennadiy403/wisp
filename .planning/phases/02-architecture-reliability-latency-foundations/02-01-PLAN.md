## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem â†’ process | `~/.config/govori/config.yaml` is user-writable; malformed or hostile YAML must not crash before the validation layer can emit a clean error |
| Filesystem â†’ process | `~/.config/govori/govori.log` is user-writable; transcripts and PII could leak via log content |
| OS â†’ process | SIGINT / SIGTERM arrive asynchronously; signal handlers run between bytecode instructions and must not deadlock |
| process â†’ filesystem | Log file directory may not exist on first run; failure to create it must not silently disable logging |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | `logging_setup.py` + every `logger.info(transcript)` callsite | mitigate | Transcript body lines are emitted at INFO level with `extra={"event":"transcript"}`. File sink is at the user's home dir with default 644 perms â€” same trust level as `~/.bash_history`. **No additional redaction** in Phase 2 (transcripts are explicitly the value of the tool; a user logging their own dictation is expected). Document this in privacy notice update later. PII filter / redaction is a v2 candidate (FEAT-04). For Phase 2: ensure CONFIG_DIR is `mkdir(parents=True, exist_ok=True)` BEFORE adding the file sink so we never fall back to stderr-only and silently leak transcripts to a parent process's tty. |
| T-02-02 | Denial of Service | `macos.perform_shutdown` / signal handler | mitigate | Signal handler does NOT acquire locks or perform I/O â€” it only sets `_shutdown_event` (RESEARCH lines 540â€“544, Pitfall 7). Logger queue is drained via `logger.complete()` before `NSApp.terminate_(None)`. |
| T-02-03 | Denial of Service | `config.load_config` on malicious YAML | mitigate | YAML is parsed with `yaml.safe_load` (not `yaml.load`) â€” no arbitrary object construction.
