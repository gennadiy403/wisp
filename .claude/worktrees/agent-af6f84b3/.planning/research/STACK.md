# Technology Stack

**Project:** Govori (macOS voice dictation tool) -- Market Readiness
**Researched:** 2026-04-17

## Current State

Single-file Python app (`govori.py`, ~72KB), Python 3.14, no `pyproject.toml`, bare `requirements.txt` with unpinned dependencies. Distributed as "clone the repo and run."

## Recommended Stack

### Build System & Packaging

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| uv | >=0.11 | Package manager, venv, build, publish | Industry standard for Python in 2026. Replaces pip, pip-tools, pipx, poetry, pyenv in one tool. 10-100x faster than pip. Cross-platform lockfile via `uv.lock`. | HIGH |
| hatchling | 1.29.0 | Build backend | Lightweight, fast, PEP 621 compliant. Better than setuptools for new projects (less config). uv supports it natively. Requires Python >=3.10 which is fine (project uses 3.14). | HIGH |
| pyproject.toml | -- | Project metadata | The only correct answer in 2026. `setup.py` is legacy. PEP 621 standard. Single source of truth for metadata, deps, scripts, tool config. | HIGH |

### Distribution Channels

| Channel | Priority | Purpose | Why | Confidence |
|---------|----------|---------|-----|------------|
| PyPI (`pip install govori`) | P0 | Primary distribution | Lowest friction for Python users. `uv tool install govori` / `pipx install govori` gives isolated CLI. | HIGH |
| Homebrew tap (`brew install gennadiy403/govori/govori`) | P1 | macOS-native distribution | macOS users expect `brew install`. Personal tap (not homebrew-core) avoids review overhead. PyObjC deps need special handling in formula. | MEDIUM |
| GitHub Releases | P2 | Source + changelog | Standard for open-source. Attach sdist/wheel to releases. | HIGH |

**NOT recommended:**
- `.dmg` / `.app` bundle -- Overkill for a CLI tool. Adds signing/notarization complexity ($99/yr Apple Developer Program) with no UX benefit for a terminal tool.
- `snap` / `flatpak` -- Linux only, irrelevant for macOS-only tool.
- PyInstaller / Nuitka binary -- Tempting but PyObjC + native audio deps make frozen binaries fragile. Not worth the debugging cost.

### Dependency Pinning

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| uv.lock | -- | Reproducible lockfile | Generated by `uv lock`. Pins exact versions + hashes. Cross-platform. Checked into git. | HIGH |

**Pinned dependency versions (pyproject.toml ranges):**

```toml
[project]
dependencies = [
    "openai>=2.30,<3",
    "anthropic>=0.90,<1",
    "sounddevice>=0.5,<1",
    "soundfile>=0.13,<1",
    "numpy>=2.0,<3",
    "av>=14,<15",
    "pyyaml>=6.0,<7",
    "pyobjc-framework-Cocoa>=12.0,<13",
    "pyobjc-framework-Quartz>=12.0,<13",
]
```

Rationale: Use compatible release ranges (`>=X,<Y`) in pyproject.toml for flexibility, but lock exact versions in `uv.lock` for reproducibility. PyObjC pinned to major 12 because major bumps can break Cocoa bindings.

### Logging

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| loguru | 0.7.3 | Application logging | Zero-config structured logging with rotation, retention, and formatting out of the box. One import (`from loguru import logger`), no handler/formatter boilerplate. Perfect for a single-file app migrating from print/stdout. structlog is better for large services; loguru wins for CLI tools. | HIGH |

Configuration pattern:
```python
from loguru import logger
import sys

# Remove default stderr handler
logger.remove()
# File logging with rotation
logger.add(
    Path.home() / ".config/govori/logs/govori.log",
    rotation="5 MB",
    retention="7 days",
    level="INFO",
    format="{time:YYYY-MM-DD HH:mm:ss} | {level: <8} | {message}",
)
# Optional stderr for --verbose
if verbose:
    logger.add(sys.stderr, level="DEBUG")
```

**NOT recommended:**
- stdlib `logging` -- Too much boilerplate for a CLI tool. Handlers, formatters, propagation -- all unnecessary complexity.
- structlog -- Great for microservices with JSON log pipelines. Overkill for a macOS CLI writing to a local file.

### Testing

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| pytest | 9.0.3 | Test framework | De facto standard. Rich plugin ecosystem. Fixtures for mocking audio/API. | HIGH |
| pytest-mock | latest | Mocking | Thin wrapper around unittest.mock with pytest fixtures. Essential for mocking OpenAI/Anthropic API calls. | HIGH |
| pytest-cov | latest | Coverage | Coverage reporting integrated with pytest. | HIGH |

Testing strategy for this project:
- **Unit tests**: Config loading, hallucination filter, silence detection, YAML parsing -- pure functions, easy to test.
- **Integration tests**: Mock OpenAI/Anthropic SDK calls, verify transcription pipeline without network.
- **Smoke tests**: Launch app with `--dry-run` flag (to be added), verify it starts and exits cleanly.
- **Skip macOS-specific tests in CI**: CGEventTap/Cocoa tests need macOS runners. Use `@pytest.mark.skipif(sys.platform != "darwin")`.

**NOT recommended:**
- unittest -- Verbose, class-based. pytest runs unittest tests anyway if needed.
- hypothesis -- Property-based testing is overkill for this project scope.

### Error Reporting (Crash Reporting)

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| sentry-sdk | 2.58.0 | Crash reporting | Industry standard. Free tier (5K errors/month) is more than enough. Python 3.6-3.14 support confirmed. Opt-in with `SENTRY_DSN` env var for privacy. | MEDIUM |

**Implementation approach:**
```python
import sentry_sdk
dsn = os.environ.get("SENTRY_DSN")  # Opt-in only
if dsn:
    sentry_sdk.init(dsn=dsn, traces_sample_rate=0)
```

**Privacy consideration:** Voice dictation tool -- users are privacy-sensitive. Sentry MUST be opt-in, not default. Document clearly in privacy notice. Never send audio data or transcription text to Sentry.

**Alternative considered:** Skip crash reporting entirely for v1.0 and rely on loguru file logs + GitHub Issues. This is the simpler path and may be sufficient for initial release. Sentry can be added in a later phase when user base grows.

### Code Quality

| Technology | Version | Purpose | Why | Confidence |
|------------|---------|---------|-----|------------|
| ruff | 0.15.11 | Linter + formatter | Replaces flake8, isort, black, pyflakes, and more. Single tool, Rust-fast. Configured in pyproject.toml. | HIGH |

```toml
[tool.ruff]
target-version = "py314"
line-length = 100

[tool.ruff.lint]
select = ["E", "F", "W", "I", "UP", "B", "SIM"]
```

**NOT recommended:**
- black + flake8 + isort -- Three tools replaced by one (ruff). No reason to use the old trio in 2026.
- pylint -- Slow, noisy, fights you on style. Ruff covers the useful rules.
- mypy -- Type checking is valuable but the codebase has zero type annotations. Adding mypy to a 1900-line untyped file is a separate milestone, not part of hardening.

### Auto-Updates

**Recommendation: Do NOT build auto-update for v1.0.**

Rationale:
- PyPI distribution handles updates via `pip install --upgrade govori` or `uv tool upgrade govori`.
- Homebrew handles updates via `brew upgrade govori`.
- Building a custom auto-updater adds significant complexity (checking versions, downloading, replacing, handling permissions) for minimal benefit at this scale.
- If needed later, the simplest approach is a `govori update` command that shells out to `pip install --upgrade govori`.

**NOT recommended:**
- pyupdater -- Abandoned/unmaintained.
- Custom update mechanism -- Complexity not justified for a CLI tool distributed via package managers.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Build backend | hatchling | setuptools | More config, legacy patterns, slower |
| Build backend | hatchling | flit | Less flexible, fewer features |
| Build backend | hatchling | uv_build | Too new (stable Jul 2025), less ecosystem adoption |
| Package manager | uv | poetry | Slower, heavier, lock format not cross-platform |
| Package manager | uv | pip + pip-tools | Two tools instead of one, slower |
| Logging | loguru | stdlib logging | Too much boilerplate for CLI |
| Logging | loguru | structlog | Overkill structured logging for local file output |
| Linting | ruff | black+flake8 | Three tools → one, slower |
| Distribution | PyPI + brew tap | PyInstaller | Fragile with PyObjC native deps |
| Distribution | PyPI + brew tap | .dmg | Overkill for CLI, requires Apple signing |

## Installation Commands

```bash
# Development setup
uv sync --dev

# Run in development
uv run govori

# Build for distribution
uv build

# Publish to PyPI (via trusted publishers, or with token)
uv publish

# End user installation
uv tool install govori    # preferred 2026 way
pipx install govori       # also works
pip install govori         # classic
brew install gennadiy403/govori/govori  # macOS native (requires tap setup)
```

## Homebrew Tap Notes

Creating a Homebrew formula for a Python tool with PyObjC dependencies requires:
1. A personal tap repo: `github.com/gennadiy403/homebrew-govori`
2. Use `homebrew-pypi-poet` to generate resource stanzas for all deps
3. PyObjC wheels are macOS-only, which is fine since this is a macOS-only tool
4. Formula should use `virtualenv_install_with_resources` pattern
5. Test block: `system bin/"govori", "--version"`

This is P1 priority -- get PyPI working first, then add the tap.
